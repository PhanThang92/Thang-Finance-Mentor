Please upgrade the existing admin CMS for “Phan Van Thang SWC” by adding an internal analytics / tracking dashboard.

IMPORTANT:
- Write implementation and code in English.
- Keep all visible UI text in Vietnamese.
- Build on top of the current admin CMS, Supabase/Postgres, and public website architecture.
- Do NOT redesign the public site.
- Do NOT overbuild a complex enterprise analytics system.
- Focus on a practical, maintainable internal dashboard that gives useful content and website performance insights.

GOAL:
Create an analytics / tracking system inside the admin CMS so the site owner can understand how the website content is performing.

This system should help answer questions such as:
- Which articles are getting the most views?
- Which videos are getting the most clicks?
- Which topics/series are performing best?
- Which homepage sections / CTA links are being clicked?
- What content is new but already getting attention?
- What should be promoted more?

The dashboard should be useful for:
- content decisions
- homepage decisions
- SEO/content strategy decisions
- brand/content growth decisions

1. FEATURE SCOPE

================================
2. WHAT TO TRACK FIRST
================================

Please track the following events first:

A. Article page view
When a user visits an article detail page

B. Video click
When a user clicks a video card or “Xem video” link

C. Article click
When a user clicks from listing/homepage into an article

D. Homepage CTA click
Examples:
- Khám phá bài viết
- Tìm hiểu thêm
- Xem kênh YouTube
- Liên hệ kết nối
- Xem thư viện video

E. Knowledge hub / topic / series click
When a user clicks into:
- Chủ đề
- Series nổi bật
- Kiến thức sections

Optional but useful:
F. Contact CTA click
G. External link click
H. Navbar item click

Requirements:
- keep tracked events practical
- prioritize meaningful business/content signals
- avoid noisy tracking that creates too much useless data

================================
3. TRACKING DATA MODEL
================================

Create a clean analytics event model in the database.

Suggested table:
- `analytics_events`

Recommended fields:
- id (uuid, primary key)
- event_type (text)
- entity_type (text, nullable)
- entity_id (uuid or text, nullable)
- entity_slug (text, nullable)
- event_label (text, nullable)
- page_path (text)
- referrer (text, nullable)
- user_agent (text, nullable, optional)
- session_id (text, nullable)
- visitor_id (text, nullable, optional)
- metadata (jsonb, optional)
- created_at (timestamp)

Examples:
- event_type = "article_view"
- entity_type = "article"
- entity_slug = "dau-tu-dai-han-khong-bat-dau-tu-von-lon..."
- page_path = "/bai-viet/..."

Requirements:
- schema should be flexible enough for multiple event types
- keep it lightweight
- preserve future expansion possibilities
- avoid making the analytics model too rigid
